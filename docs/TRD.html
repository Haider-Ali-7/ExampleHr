1 Executive Summary

The HCM Leave Management System is a full-stack web application that enables employees to request time off, managers to approve or reject requests, and administrators to manage employee balances across multiple locations. The system emphasizes real-time synchronization, optimistic user experiences, and role-based access control.

6 Optimistic Updates & Background Refresh

Why Optimistic Updates?

Instant Feedback

Users see changes immediately without waiting for network roundtrips. The UI responds in under 16ms (single frame) instead of 200-500ms typical API latency.

Success is the Common Case

In a well-designed system, 95%+ of user actions succeed. Optimistic updates optimize for this majority while gracefully handling the rare failure.

Automatic Rollback

When errors occur, the UI seamlessly reverts to the previous state. Users see a clear error message and can retry. No manual refresh needed.

Implementation Pattern

const mutation = useMutation({
  mutationFn: submitRequest,

  onMutate: async (newData) => {
    // 1. Cancel in-flight queries
    await queryClient.cancelQueries({ queryKey });

    // 2. Snapshot current state
    const previous = queryClient.getQueryData(queryKey);

    // 3. Optimistically update cache
    queryClient.setQueryData(queryKey, (old) =>
      [...old, { ...newData, isOptimistic: true }]
    );

    return { previous }; // For rollback
  },

  onError: (err, vars, context) => {
    // Rollback to snapshot
    queryClient.setQueryData(queryKey, context?.previous);
    showToast({ type: 'error', message: err.message });
  },

  onSettled: () => {
    // Reconcile with server truth
    queryClient.invalidateQueries({ queryKey });
  },
});

Why Background Refresh with SSE?

Multi-User Consistency

When Employee A submits a request, Manager B sees it appear instantly. When Manager approves, Employee A's balance updates in real-time without polling.

Server as Source of Truth

SSE events push authoritative updates. Even if optimistic state diverges temporarily (e.g., concurrent approvals), the server reconciles the final state.

Efficient Bandwidth

Unlike polling, SSE only transmits data when changes occur. A single persistent connection replaces repeated HTTP requests, reducing server load by 90%+.

Stale-While-Revalidate Pattern

Show cached data instantly
Revalidate in background

SSE Event Flow

1 Server action triggers broadcast(event)
2 EventEmitter emits to all SSE streams
3 useLeaveSSE receives event
4 Hook invalidates relevant query keys
5 TanStack Query refetches fresh data

7 Challenges Faced & Solutions Adopted

1 SSE State Updates Violating React Hooks Rules

Challenge

Both useAdminLeave and useManagerLeave hooks call setState inside useEffect when SSE events arrive, triggering ESLint's react-hooks/set-state-in-effect warning. 

// ESLint complains about this pattern
useEffect(() => {
  if (lastEvent?.type === 'balance_updated') {
    setStaleCells(prev => /* violation */
      new Set(prev).add(staleKey(...))
    );
  }
}, [lastEvent]);

Solution

Intentional deviation with documentation. The SSE event-driven pattern requires synchronous state updates to prevent race conditions between optimistic updates and server events. We use a targeted ESLint disable with inline comment explaining the rationale. 

// eslint-disable-next-line react-hooks/set-state-in-effect
// Intentional: SSE events require immediate state sync
// to mark cells stale before user interaction
setStaleCells(prev => new Set(prev).add(staleKey(...)));

2 Balance Overselling on Concurrent Requests

Challenge

If an employee with 5 annual days submits two 3-day requests simultaneously, both might pass validation if each only checks the static balance (5 days), resulting in 6 days approved against 5 available.

Race Condition Scenario:
T1: Request A checks balance (5) -> passes
T2: Request B checks balance (5) -> passes
T3: Both approved -> 6 days deducted from 5

Solution

Optimistic Reservation Pattern. Available balance is calculated as: available = balance - pendingDays. All pending requests of the same type/location/employee are summed before validation. 

const pendingDays = await getPendingDays(
  employeeId, location, leaveType
);
const available = balance.balance - pendingDays;

if (days > available) {
  return NextResponse.json({
    error: 'Insufficient balance',
    code: 'INSUFFICIENT_BALANCE',
    available,
  }, { status: 422 });
}

3 Form Reset After Successful Submission

Challenge

How does the LeaveRequestForm component know when to reset after a successful submission? Callbacks create coupling; polling mutation state is inefficient.

Requirements:
  • Reset form fields after success
  • Preserve location selection (frequently reused)
  • No callback prop drilling
  • No stale closure bugs

Solution

Reactive Counter Pattern. The hook exposes a submitCount number that increments on each successful submission. The form component watches this with useEffect and resets when it changes. 

// In hook: increment on success
const submitCountRef = useRef(0);
onSuccess: () => {
  submitCountRef.current += 1;
  setSubmitCount(submitCountRef.current);
}

// In form: react to changes
useEffect(() => {
  if (submitCount === 0) return; // Initial
  resetForm({ preserveLocation: true });
}, [submitCount]);

4 SSE Duplicate Event Processing

Challenge

Network hiccups cause EventSource reconnections. The server may re-send recent events during reconnection, leading to duplicate toasts, double cache invalidations, or stale marker flicker.

Symptoms:
  • Multiple "New request" toasts for same request
  • Unnecessary network refetches
  • UI flicker from repeated state updates

Solution

Deduplication via Event ID Set. Each broadcast event includes a unique UUID. The client maintains a seenIdsRef Set and skips events already processed. 

const seenIdsRef = useRef<Set<string>>(new Set());

eventSource.onmessage = (e) => {
  const parsed = JSON.parse(e.data);

  // Skip if already processed
  if (parsed.id && seenIdsRef.current.has(parsed.id)) {
    return;
  }

  // Mark as seen
  if (parsed.id) {
    seenIdsRef.current.add(parsed.id);
  }

  setLastEvent(parsed);
};

5 Admin Cell Staleness Detection

Challenge

The admin views a table of employee balances. When a manager approves a request (balance changes), the specific cell showing that employee's balance is now stale. How do we indicate this without a full page refresh?

Cell Identification:
employeeId + location + leaveType = unique cell

Solution

Stale Cell Set with Composite Keys. A staleCells Set stores composite keys. SSE events add keys; manual updates or refetches clear them. Cells render an amber indicator when their key is present. 

function staleKey(empId, loc, type) {
  return `${empId}:${loc}:${type}`;
}

// On SSE balance_updated event:
setStaleCells(prev =>
  new Set(prev).add(staleKey(emp, loc, type))
);

// In BalanceCell render:
{staleCells.has(staleKey(...)) && (
  <span className="amber-dot animate-pulse" />
)}

6 Testing Hooks with Complex Dependencies

Challenge

Hooks like useEmployeeLeave depend on TanStack Query, SSE, Zustand, and browser APIs. Full integration testing requires complex environment setup that's fragile and slow.

Dependencies:
  • QueryClientProvider wrapper
  • EventSource polyfill
  • localStorage mock
  • Fetch mock with timing

Solution

Contract-Based Testing. Instead of testing the entire hook, we extract and test the mutationFn (API contract) in isolation. Mock fetch to verify request shape, test error paths for each error code. 

// Test the API contract, not the hook
it('sends all required fields', async () => {
  const fetchSpy = vi.spyOn(global, 'fetch');
  await submitRequestApi(testData);

  const body = JSON.parse(fetchSpy.mock.calls[0][1].body);
  expect(body).toHaveProperty('reason'); // Regression
});
